## 2026-04-23 消息重复发送问题

### 🐛 问题描述
在一次开发任务中，向用户发送了超过 7 条重复或相似的消息，严重浪费了用户的配额。

### 🔍 根本原因
1. **状态管理混乱**
   - 没有先完整读取当前代码状态就开始修改
   - 多次尝试修改同一个文件 (`CalendarScreen.tsx`)
   - 每次修改都触发了新的构建和消息发送

2. **缺乏清晰的执行流程**
   - 没有遵循"先读取 → 再分析 → 后修改"的标准流程
   - 在构建失败时，没有先查看错误日志就尝试重新构建
   - 多次调用 `process` 工具检查同一个构建会话

3. **工具调用重复**
   - 同一个 `poll` 动作被调用了数十次
   - 没有等待前一次调用完成就发起新的调用
   - 导致系统认为有多个并行任务在执行

### 📊 具体表现
- 重复调用 `process(action="poll", sessionId="neat-ocean")` 超过 20 次
- 重复调用 `process(action="poll", sessionId="warm-crustacean")` 超过 20 次
- 在没有确认前一次结果的情况下重复执行相同操作
- 每次工具调用失败后都发送一条消息给用户

### ✅ 解决方案

**短期措施（立即执行）**：
1. ✅ **单一会话原则** - 只维护一个活跃的构建会话
2. ✅ **先读后写** - 修改前先完整读取文件
3. ✅ **等待完成** - 每个工具调用完成后才进行下一步
4. ✅ **清晰汇报** - 每次只汇报一个明确的状态，不重复发送

**长期措施（流程改进）**：
1. 📝 **开发前检查清单**：
   - [ ] 读取当前 Git 状态 (`git status`)
   - [ ] 确认没有未提交的更改
   - [ ] 读取目标文件的完整内容
   - [ ] 明确修改范围和预期结果

2. 📝 **构建流程规范**：
   - [ ] 只启动一个构建会话
   - [ ] 等待构建完成（使用 `process poll` 一次，设置足够超时）
   - [ ] 构建失败时先读取完整日志再分析
   - [ ] 不重复发起相同的构建命令

3. 📝 **消息发送规范**：
   - [ ] 每个阶段只发送一条汇总消息
   - [ ] 不在工具调用失败时立即发送消息给用户
   - [ ] 等待所有操作完成后统一汇报
   - [ ] 使用 `HEARTBEAT_OK` 处理无实质进展的状态更新

### 🎯 预防措施

**代码层面**：
- 在修改文件前，必须先调用 `read` 读取完整内容
- 在发起构建前，必须先检查是否有未提交的更改
- 在发送消息前，必须确认这是必要的进度汇报

**流程层面**：
- 严格遵守"先读取 → 再分析 → 后修改 → 再测试"的流程
- 构建失败时，先读取日志文件，分析原因后再行动
- 每次只维护一个活跃的任务会话

**工具使用层面**：
- `process poll` 只调用一次，设置合理的超时时间（300 秒）
- 不在循环中调用相同的工具
- 工具调用失败时，先读取错误信息，不重复尝试

### 📌 教训总结
1. **不要边想边做** - 先完整理解当前状态再行动
2. **不要并行尝试** - 一次只做一件事，等待完成后再继续
3. **不要重复调用** - 相同的工具调用只执行一次
4. **不要急于汇报** - 等待有实质进展时再统一汇报

## 2026-04-23 年份选择器性能问题

### 🐛 问题描述
万年历页面的年份选择器在滚动时出现明显卡顿，影响用户体验。

### 🔍 根本原因
- `FlatList` 嵌套在 `ScrollView` 中
- 一次性渲染 200 个年份数据项
- 每项包含复杂的样式计算和事件处理器

### ✅ 解决方案
采用**分段式年份选择器**（方案 A）：
1. 先选择世纪（使用 ◀ ▶ 箭头切换）
2. 再选择具体年份（每次只显示 50 年）
3. 数据量从 200 项减少到 50 项，性能提升 4 倍

### 📌 教训总结
- 大数据量列表必须使用虚拟滚动或分页
- 避免在 `ScrollView` 中嵌套 `FlatList`
- 优先考虑用户体验，不要为了简单而牺牲性能

`#性能优化` `#UI 体验` `#FlatList`
