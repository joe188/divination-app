## ⚠️ 重要教训：依赖版本问题的折腾

今天花了很长时间在依赖软件版本问题上折腾，这是一个重要的教训！

### 问题背景
- 构建时遇到各种依赖版本冲突
- Gradle 版本、Kotlin 版本、Android SDK 版本不匹配
- 第三方库版本与 React Native 版本不兼容

### 折腾过程
1. 尝试升级 Gradle 版本 → 失败
2. 尝试降级 Kotlin 版本 → 失败
3. 尝试修改第三方库版本 → 失败
4. 最后发现：**不要轻易改动依赖版本！**

### 关键经验
1. ✅ **保持现有依赖版本不变** - 如果能构建成功，就不要动
2. ✅ **不要盲目升级依赖** - 升级可能带来新的兼容性问题
3. ✅ **不要盲目降级依赖** - 降级可能导致功能缺失
4. ✅ **遇到问题时先检查配置** - 不是版本问题，可能是配置问题
5. ✅ **使用 `./gradlew clean` 清理缓存** - 很多时候是缓存问题

### 正确的做法
```bash
# 遇到构建问题时，按顺序尝试：
1. ./gradlew clean                    # 清理缓存
2. ./gradlew assembleRelease          # 重新构建
3. 如果还是失败，检查错误日志        # 不要急着改版本
4. 如果确定是版本问题，再考虑升级/降级 # 但要谨慎
```

- 编译所有模块的 Kotlin/Java 代码
- 生成 DEX 文件
- 合并 DEX 文件

#### 阶段 4：资源优化和打包（约 1-2 分钟）
- 合并资源
- 压缩资源
- 优化 PNG 图片
- 打包 APK
- 对齐 APK（zipalign）
- 签名 APK（使用 debug keystore）

### 六、常见问题和解决方案

#### 问题 1：构建失败 - Execution failed for task ':app:processReleaseResources'
- **原因**：资源文件有问题
- **解决**：检查 `android/app/src/main/res/` 下的资源文件

#### 问题 2：构建失败 - Execution failed for task ':app:compileReleaseKotlin'
- **原因**：Kotlin 代码有语法错误
- **解决**：运行 `npx tsc --noEmit` 检查 TypeScript 错误

#### 问题 3：APK 安装失败 - INSTALL_FAILED_UPDATE_INCOMPATIBLE
- **原因**：设备上已有相同包名但签名不同的应用
- **解决**：先卸载旧版本 `adb uninstall com.lingshupaipan`

#### 问题 4：APK 闪退
- **原因**：运行时错误
- **解决**：查看日志 `adb logcat | grep -i "lingshupaipan"`

#### 问题 5：构建时间过长
- **原因**：首次构建需要下载依赖
- **解决**：后续构建会快很多（约 5-8 分钟）

## 🔬 深入技术细节

### 一、JS Bundle 打包的详细过程

**Metro 打包器的工作原理**：
```
入口文件 (index.js)
    ↓
解析依赖关系 (Dependency Resolution)
    ↓
转换代码 (Transformation)
    - TypeScript → JavaScript
    - JSX → JavaScript
    - 应用 Babel 插件
    ↓
优化代码 (Optimization)
    - Tree Shaking
    - 代码压缩
    ↓
生成 Bundle (index.android.bundle)
```

**关键配置文件**：
- `metro.config.js` - Metro 配置
- `babel.config.js` - Babel 转换配置
- `tsconfig.json` - TypeScript 配置

### 二、CMake 编译的详细过程

**CMake 构建流程**：
```
CMakeLists.txt (配置文件)
    ↓
生成构建系统 (Generate Build System)
    - 检测编译器
    - 配置编译选项
    ↓
编译原生代码 (Compile Native Code)
    - C/C++ → Object Files (.o)
    - 链接 → Shared Libraries (.so)
    ↓
为每个架构编译 (Build for Each ABI)
    - arm64-v8a
    - armeabi-v7a
    - x86
    - x86_64
```

**编译时间分析**：
- Hermes 编译：约 2-3 分钟（最大的库）
- op-sqlite 编译：约 30 秒
- react-native-screens 编译：约 1 分钟
- react-native-reanimated 编译：约 1 分钟

### 三、DEX 打包的原理

**DEX 文件是什么**：
- DEX (Dalvik Executable) 是 Android 的可执行文件格式
- 包含编译后的 Java/Kotlin 字节码
- Android 5.0+ 使用 ART (Android Runtime) 运行 DEX

**DEX 打包流程**：
```
Java/Kotlin 源代码
    ↓
编译为 .class 文件 (javac/kotlinc)
    ↓
转换为 .dex 文件 (d8/r8)
    ↓
合并所有 .dex 文件 (mergeDex)
    ↓
优化 DEX 文件 (optimizeDex)
    ↓
打包到 APK (packageApk)
```

### 一、实际遇到的问题和解决过程

**问题 1：构建卡在某个任务不动**
- **现象**：构建过程卡在 `:app:processReleaseResources` 不动
- **原因**：资源文件过大或损坏
- **解决**：检查并压缩资源文件

**问题 2：Kotlin 编译警告太多看不清**
- **现象**：大量 deprecation warnings 淹没了真正的错误
- **原因**：第三方库使用了过时的 API
- **解决**：忽略警告或更新第三方库

**问题 3：APK 安装后闪退**
- **现象**：APK 安装成功，但启动后立即闪退
- **原因**：JS Bundle 打包失败或原生库缺失
- **解决**：查看日志 `adb logcat | grep -i "lingshupaipan"`

### 二、构建过程中的坑和注意事项

**坑 1：不要在构建过程中修改代码**
- **后果**：可能导致构建失败或生成的 APK 不一致

**坑 2：不要忽略 TypeScript 错误**
- **后果**：可能导致运行时错误

**坑 3：不要使用 `--warning-mode all`**
- **后果**：会显示大量警告，淹没真正的问题

**坑 4：不要忘记清理缓存**
- **后果**：可能导致构建失败或使用旧的代码
- **建议**：遇到奇怪问题时，先清理缓存：`./gradlew clean`

**成功标志**：

**检查清单**：

## 🔬 深入技术细节

### 一、JS Bundle 打包的详细过程

**Metro 打包器的工作原理**：
```
入口文件 (index.js)
    ↓
解析依赖关系 (Dependency Resolution)
    ↓
转换代码 (Transformation)
    - TypeScript → JavaScript
    - JSX → JavaScript
    - 应用 Babel 插件
    ↓
优化代码 (Optimization)
    - Tree Shaking
    - 代码压缩
    ↓
生成 Bundle (index.android.bundle)
```

**关键配置文件**：
- `metro.config.js` - Metro 配置
- `babel.config.js` - Babel 转换配置
- `tsconfig.json` - TypeScript 配置

### 二、CMake 编译的详细过程

**CMake 构建流程**：
```
CMakeLists.txt (配置文件)
    ↓
生成构建系统 (Generate Build System)
    - 检测编译器
    - 配置编译选项
    ↓
编译原生代码 (Compile Native Code)
    - C/C++ → Object Files (.o)
    - 链接 → Shared Libraries (.so)
    ↓
为每个架构编译 (Build for Each ABI)
    - arm64-v8a
    - armeabi-v7a
    - x86
    - x86_64
```

**编译时间分析**：
- Hermes 编译：约 2-3 分钟（最大的库）
- op-sqlite 编译：约 30 秒
- react-native-screens 编译：约 1 分钟
- react-native-reanimated 编译：约 1 分钟

### 三、DEX 打包的原理

**DEX 文件是什么**：
- DEX (Dalvik Executable) 是 Android 的可执行文件格式
- 包含编译后的 Java/Kotlin 字节码
- Android 5.0+ 使用 ART (Android Runtime) 运行 DEX

**DEX 打包流程**：
```
Java/Kotlin 源代码
    ↓
编译为 .class 文件 (javac/kotlinc)
    ↓
转换为 .dex 文件 (d8/r8)
    ↓
合并所有 .dex 文件 (mergeDex)
    ↓
优化 DEX 文件 (optimizeDex)
    ↓
打包到 APK (packageApk)
```

- **现象**：APK 安装成功，但启动后立即闪退

- **解决**：查看日志 `adb logcat | grep -i "lingshupaipan"`

## 🐛 晚间修复：BaZiInputScreen 布局问题

### 时间：2026-04-19 23:42

### 问题描述
八字排盘功能点击"开始排盘"按钮后报错：
```
TypeError: Cannot read property 'year' of undefined
```

### 问题诊断过程

**1. 初步诊断**：
- 以为是 ResultScreen.tsx 的数据传递问题
- 修改了 ResultScreen.tsx 的数据处理逻辑
- 但错误仍然存在

**2. 深入诊断**：
- 使用 `adb shell uiautomator dump` 获取 UI 层次结构
- 发现"开始排盘"按钮在屏幕外面：
  ```
  content-desc: 🔮 开始排盘
  boundsInScreen: Rect(53, 2691 - 1028, 2400)
  visible: false
  ```
- 屏幕高度是 2400，但按钮的 Y 坐标是 2691，超出了屏幕

**3. 根本原因**：
- 之前优化时把所有屏幕的 ScrollView 替换成了 View
- 但 BaZiInputScreen 包含很多输入项（出生时间、地点、真太阳时校正等）
- 内容超出了屏幕高度，导致"开始排盘"按钮在屏幕外面
- 用户无法看到或点击按钮

### 解决方案

**修复 BaZiInputScreen.tsx**：
- 恢复使用 ScrollView
- 让用户可以滚动查看所有内容

**修改内容**：
```tsx
// 之前（错误）：
<View style={styles.content}>
  {/* 所有内容 */}
</View>

// 修复后（正确）：
<ScrollView style={styles.content} contentContainerStyle={styles.scrollContent}>
  {/* 所有内容 */}
</ScrollView>
```

### JS Bundle 打包问题

**问题**：
- Gradle 的 `assembleDebug` 任务不会自动重新打包 JS Bundle

```bash
  --bundle-output android/app/src/main/assets/index.android.bundle \
  --assets-dest android/app/src/main/res

# 然后重新构建 APK
cd android && ./gradlew assembleDebug
```
